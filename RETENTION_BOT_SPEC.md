## 1. Обзор и границы

Расширение добавляет к существующему support-chat второй канал — Telegram-бота с
ретеншн-персоной Никой — **переиспользуя то же AI-ядро** (`chat_service`, `prompts`,
`openai_client`, настройки, секреты, мультитенантность). Транспорт другой (Telegram
webhook вместо виджета), поведенческий режим другой (флирт + фото вместо строгой
поддержки), но мозг один.

**Что строим**

- **Telegram-бот** на продукт: приветствие, гейт подписки на канал, меню, живая AI-переписка Ники.
- **Отдельную retention-KB**: единую плоскую базу сценариев/офферов/линков (support-KB на 7 разделов сюда НЕ грузится — см. §4).
- **Медиатеку**: до ~500 фото на бренд с описаниями, тегами, уровнями (VIP × ступень откровенности), трекингом просмотров.
- **Роутинг на живых менеджеров**: пул Telegram-аккаунтов, равномерная раздача.
- **Таргетинг по профилю игрока**: связка Telegram-игрок ↔ аккаунт на сайте + синхронизация всего профиля.
- **Админку**: retention-KB, медиатека, менеджеры, ретеншн-персона, аналитика — в текущей SPA.

**Чего НЕ строим**

- **AI-поддержку в Telegram не дублируем** (решение 8) — поддержка остаётся на сайте. В Telegram AI = только ретеншн-Ника; «человеческая поддержка» = передача менеджеру.
- **Комбинированного «ретеншн + поддержка» бота НЕ строим.** Ника занимается *только* ретеншном. Любой разговор о поддержке, жалобе, блоке аккаунта или просьбе о живом человеке она **не обрабатывает**, а роутит наружу (§3, §4). Никакого support-KB, топик-роутинга и решения вопросов казино внутри бота.
- **Органического входа нет** — бот открывается только с сайта, всегда с параметрами игрока (§2). Из канала бота открыть нельзя.
- **Автоподписки нет** — платформа Telegram её запрещает; заменяется гейтом `getChatMember`.

> **Инвариант.** Все существующие инварианты support-chat остаются в силе: byte-stable
> Layer-1 в пределах продукта, атомарная запись хода, каждый OpenAI-вызов →
> `ai_interaction_logs`, каждая per-turn/per-session строка несёт `product_id`, доступ к
> БД только через `db.*`, секреты через `secretbox`.

## 2. Воронка и вход в бота

Единственный источник входа — **сайт**. Обе точки входа генерят per-user диплинк и
**всегда несут профиль игрока** (как handshake в поддержке) — не только VIP, а весь набор
данных для таргетинга (§8). Различает входы только флаг эскалации, который меняет меню.

| Вход | Откуда | Данные | Меню после гейта |
|---|---|---|---|
| **Ретеншн-кнопка** | кнопка «Написать Нике» на сайте | id, продукт, профиль (`_CONTEXT_FIELDS`: vip_level, balance, country…) | `[💬 Ника]` |
| **Эскалация** | кнопка эскалации в чате поддержки | то же **+ флаг escalation** | `[👩 К менеджеру]` + `[💬 Ника]` |

### Диплинк и одноразовый nonce

Полный handshake не влезает в `start=` (лимит 64 симв.), поэтому обмен через nonce:

```
# 1. сайт → наш бэкенд (handshake игрока)
POST /api/retention/deeplink  { handshake, escalation: true|false }
   → { nonce: "r7Kx…20симв", deep_link: "https://t.me/nika_casino_bot?start=r7Kx…" }

# 2. игрок жмёт кнопку → открывается бот → Start
Telegram → POST /telegram/webhook   /start r7Kx…

# 3. бот обменивает nonce → контекст игрока, помечает nonce used, фиксирует связку
   nonce одноразовый, TTL ~120 сек, при обмене создаётся/обновляется retention_users
```

**Enforcement.** Бот на `/start` **требует валидный nonce**. Нет nonce / протух / уже
использован → «Открой чат через свой кабинет на сайте», дальше не пускаем. Так
гарантируем: параметры игрока есть **всегда**, органического «тир = 1» не существует
(tier=1 — только аварийный fallback при отсутствии данных).

### Гейт подписки на канал (для всех)

После обмена nonce, до любого меню, бот проверяет подписку и не отвечает, пока игрок не подписан:

```
getChatMember(channel_id, user_id)
  ├─ не подписан → «Подпишись на канал, чтобы продолжить»
  │              [📢 Открыть канал] [✅ Я подписался]  (повторная проверка)
  └─ подписан → показываем меню по типу входа (§3)
```

Бот должен быть **админом канала** (для `getChatMember` и для публикации закреплённого
поста с кнопками). Канал = только цель подписки, входом в бота он не является.

## 3. Сценарий бота и меню

Меню зависит от типа входа (наличие флага эскалации в обменянном nonce). Кнопку «к
менеджеру» **видит только эскалированный** игрок.

```
─── ЭСКАЛИРОВАННЫЙ ───────────────────────────────────
чат поддержки на сайте → «эскалация» → deeplink(escalation=true) → Start
  → обмен nonce (знаем игрока + профиль + флаг) → фикс связки tg↔player
  → ГЕЙТ подписки
  → меню:
       [👩 Перейти к менеджеру]   ← главное, round-robin выдаёт t.me/<manager>
       [💬 Пообщаться с Никой]    ← AI-ретеншн + фото под профиль
  → менеджер = результат эскалации (живой человек)   ИЛИ   Ника = AI-диалог

─── РЕТЕНШН-КНОПКА ───────────────────────────────────
сайт «Написать Нике» → deeplink(escalation=false) → Start
  → обмен nonce (знаем игрока + профиль) → фикс связки
  → ГЕЙТ подписки
  → меню: [💬 Пообщаться с Никой]   ← только Ника
```

### Правило роутинга наружу — Ника НЕ решает вопросы

Как только в диалоге всплывает **поддержка, жалоба, блок аккаунта, ввод-вывод, просьба о
живом человеке или тема ответственной игры** (лимиты/пауза/self-exclusion) — Ника роняет
флирт и **не пытается ответить**, а перенаправляет:

- **Эскалированный вход** → кнопка `[👩 К менеджеру]`.
- **Ретеншн-вход** (менеджера в меню нет) → «эти вопросы решают в поддержке на сайте» + возврат в чат поддержки.

Единое поведение для всех таких тем — никакого разбора вопроса внутри бота. Реализуется
ретеншн-директивой Layer 1 + сентинелом `[[HANDOFF]]` (§11).

## 4. Ретеншн-Ника (AI) и изоляция баз знаний

**Ядро одно — промпт собирается по-разному.** Персона Ника (тон, «ты», лёгкий флирт, без
эмодзи, RG, links-policy из `SYSTEM_CORE`) общая, но каждый режим — это своя сборка слоёв
и **своя, изолированная база знаний**. Support и retention **не делят Layer 2**.

> **Support-KB в ретеншн НЕ грузится.** Большая support-KB (7 разделов) — контент *для
> решения вопросов казино* и относится **только** к support-режиму, где грузится блок
> выбранного топика. В retention-режиме её нет вообще: у ретеншн-бота нет топиков (нет
> пикера — игрок выбрал Нику), и грузить туда 7 разделов и не нужно, и вредно (токены + не
> та задача). Вместо неё — **отдельная единая retention-KB**.

| Слой | Support-режим (есть) | Retention-режим (новый) |
|---|---|---|
| **Layer 1** (byte-stable) | persona core + KB-grounding, escalation-restraint, greeting, формат… | persona core + **директивы вовлечения**: поддержать диалог, мягко вести к игре, дозированно предлагать фото. **Правило роутинга наружу** (§3): любую поддержку/жалобу/RG отдать через `[[HANDOFF]]`. **Нет** KB-grounding / escalation-restraint / топик-роутинга. |
| **Layer 2** (KB) | **per-topic** support-KB: блок ОДНОГО выбранного топика из 7, меняется при смене топика | **единая retention-KB** (сценарии + линки/офферы), грузится ЦЕЛИКОМ, byte-stable per product. **Support-KB сюда не попадает.** |
| **Layer 3** (per-request) | язык, персонализация, топик-роутинг, история | язык, персонализация **по полному профилю** (§8), история + **список фото-кандидатов** (§5). Топик-роутинга нет. |

### Retention-KB — устройство

- **Отдельный product-scoped стор** `retention_kb` (НЕ `kb_topics`) — плоский набор
  **сценарных записей** `{title, when/триггер, что_говорить, links}`. Это разговорные
  плейбуки/офферы, а не Q&A: «давно не заходил», «спрашивает про бонусы», «близок к
  депозиту», «хочет вывести» и т.д.
- **Грузится целиком** в Layer 2 — она компактная и вся релевантна, поэтому блок
  **byte-stable в пределах продукта** и остаётся в кэшируемом префиксе (как support-KB, но
  без переключения топиков). Если разрастётся — добавим выборку релевантных сценариев (как
  список фото-кандидатов); для v1 целиком.
- **Отдельно от описаний фото** (§5): медиатека — каталог картинок; retention-KB — что и
  как Ника говорит + какие линки даёт.
- **Единая** — все сценарии в одном месте, дописываются по мере роста. Редактируется из
  админки, раздел B, под-вкладка «Retention KB / Линки по retention» (§12).

`prompt_variables` (persona/brand/tone) остаются per-product. Технически ретеншн — вторая
сборка Layer-1 рядом с support-сборкой в `prompts.py` (напр. `build_messages(mode=...)`), с
подстановкой своей retention-KB в Layer 2; byte-stable-инвариант держится в пределах
*продукт × режим*. Язык ответа и подписей к фото — по существующей language-директиве.

## 5. Медиатека и выдача фото

Ключевая идея: игрок **переписывается** с Никой текстом, она шлёт фото. Текстовое описание
фото делает **двойную работу**.

- **Роль 1 — выбор.** Модель читает описания кандидатов и матчит запрос игрока («покажи в
  купальнике») по смыслу — **без эмбеддингов**, как топик-роутинг.
- **Роль 2 — грунтинг.** Описание возвращается модели после отправки — «ты отправила фото,
  на котором: …» — чтобы подпись Ники **совпадала с картинкой**. Модель не видит пиксели;
  описание = то, что она «видит».

### Цикл выдачи (копия машинерии топик-роутинга)

```
1. игрок: «покажи что-нибудь… на пляже»
2. бэкенд кладёт в Layer 3 СПИСОК КАНДИДАТОВ (только допустимые):
      фильтр: product_id, level_min ≤ vip_level, stage ≤ unlocked_stage,
              id NOT IN seen(user);  N = candidate_list_size (6)
      каждый: { id, stage, description, tags }
3. модель по контексту эмитит [[PHOTO:id]]  (id из списка)
4. бэкенд валидирует id по допустимому набору
   → шлёт фото в Telegram (по telegram_file_id — мгновенно)
   → пишет retention_photo_views, инкрементит счётчики
5. описание выбранного фото → обратно модели как контекст
6. Ника генерит подпись, грунтованную описанием  → текст ≈ картинка
   (подпись — caption самого фото, одним сообщением)
```

### Отбор кандидатов

Бэкенд предотбирает **по полному профилю игрока и истории** (VIP-тир — лишь один из
сигналов; также баланс, стаж, сегмент — §8): только неувиденные, из текущего
`unlocked_stage` (и на 1 ступень вперёд для «дразнилки», если позволяет таргетинг), с
разнообразием тегов относительно недавно отправленных, приоритет — под сегмент игрока. Не
весь каталог из 500 (токены) — ровно `candidate_list_size` штук.

> **file_id-кэш.** Первую отправку фото грузим с Railway Volume; Telegram возвращает
> `file_id`, сохраняем в `retention_photos.telegram_file_id` и дальше пересылаем по нему —
> без повторной загрузки и egress.

### Что в описании

- **Что на фото:** обстановка, наряд, поза, настроение — «пляж, закат, красный купальник, улыбается через плечо».
- **Теги** для матчинга: `[пляж, купальник, лето]`.
- **level_min** (VIP-тир) и **stage** (ступень откровенности).
- Описание **внутреннее** — игроку сырым не показывается; подпись Ника пишет сама.

## 6. Прогрессия, лимиты, проактивность

Хард-гейт фото — два независимых измерения: **минимальный тир** (`level_min`) × **ступень
откровенности** (`stage`: обычные → костюм → купальник → …). Поверх хард-гейта идёт
**таргетинг по полному профилю** (§8): баланс, стаж, сегмент влияют на приоритет выдачи,
темп прогрессии и тон Ники — не только VIP. Плюс дневной лимит и правила проактивности. Все
пороги — knob'ы в settings-группе `retention`.

### Ступень (stage) — как растёт

`unlocked_stage` стартует с 1. Повышение **детерминирует бэкенд**; модель может лишь подсказать.

| Триггер | Условие |
|---|---|
| Вовлечённость | накопленные осмысленные сообщения игрока ≥ `stage_advance_msgs[next]` (напр. 20 / 45 / 80) |
| Хинт модели `[[STAGE_UP]]` | принимается, только если пройден мягкий порог вовлечённости **И** тир допускает **И** прошло ≥ `stage_advance_min_hours` с прошлого повышения |
| Потолок по тиру | next_stage ≤ `max_stage_by_tier[vip_level]` — VIP гейтит самые откровенные ступени |
| Спейсинг | не более одного повышения в `stage_advance_min_hours` (по умолч. 24 ч) |

**Базовое условие для `[[STAGE_UP]]`:** модель эмитит хинт, когда *(а)* есть новое
допустимое фото следующей ступени под этого игрока, *или* *(б)* игрока надо «активировать»
(охладел / давно не отвечал) — но фактическое повышение случается только если
бэкенд-гард (порог + спейсинг + потолок тира) пропускает. Модель предлагает, бэкенд решает.

### Дневной лимит и проактивность

| Knob (группа `retention`) | Дефолт | Смысл |
|---|---|---|
| `daily_photo_cap` | 10 | фото на игрока в день (проактив + реактив), сброс в полночь |
| `proactive_photo_cooldown_msgs` | 6 | Ника сама предлагает фото не чаще 1 раза в 6 сообщений игрока |
| `candidate_list_size` | 6 | сколько кандидатов кладём в Layer 3 |
| `stage_advance_msgs` | 20/45/80 | накопленные сообщения для ступеней 2/3/4 |
| `stage_advance_min_hours` | 24 | минимум между повышениями ступени |
| `max_stage_by_tier` | map | потолок ступени по VIP-тиру |
| `nonce_ttl_sec` | 120 | жизнь одноразового диплинк-nonce |

> **Правило проактивности.** **Реактив** (игрок явно просит фото) — всегда разрешён,
> обходит cooldown, но считается в `daily_photo_cap`. **Проактив** (Ника сама) — не чаще
> `proactive_photo_cooldown_msgs`, никогда два проактива подряд без ответа игрока, всё в
> рамках дневного лимита. При исчерпании лимита Ника продолжает переписку текстом без фото.

## 7. Роутинг на менеджеров

Пул из ~5 Telegram-аккаунтов на продукт («Маша NikaBet» / `@masha_nikabet`), раздаём
**равномерно round-robin**. Общение с менеджером = результат эскалации.

- В админке (per-product): `{ display_name, username, active, assigned_count }`.
- **Round-robin** по счётчику; следующему игроку — следующий активный менеджер.
- **Липкое назначение:** первый раз round-robin, дальше `retention_users.assigned_manager_id` — вернувшийся попадает к тому же (континуити).
- Неактивного (отпуск) выключаем флагом → выпадает из ротации.
- Передача = кнопка/ссылка `t.me/<username>`; дальше личный диалог вне нашего контроля.
  Логируем только факт передачи + кому (аналитика, равномерность).

## 8. Профиль игрока и таргетинг

Таргетинг в ретеншн-чате идёт **не только по VIP-уровню**, а по всему профилю игрока.
**Набор полей = существующий whitelist** `prompts._CONTEXT_FIELDS` — тот самый, что уже
настраивается в админке как тест-профиль. Ничего нового не изобретаем; профиль приходит в
handshake (в dev — из тест-профиля) и синхронизируется как единое целое.

### Поля профиля = `_CONTEXT_FIELDS` (снимок в `retention_users`)

| Поле | Применение в таргетинге |
|---|---|
| `vip_level` | хард-гейт фото (`level_min`) + потолок ступени |
| `balance` | «сколько денег» — приоритет внимания, тон, темп прогрессии |
| `registration_date` | стаж игрока (новичок vs ветеран) — сегментация / реактивация |
| `activation_status` | активирован ли — влияет на подачу |
| `country` | сегмент / персонализация реплик |
| `full_name` | персонализация (имя в приветствии) |
| `id`, `email` | идентификация / связка (в таргетинге не участвуют) |

Профиль уходит и в Layer 3 персонализации, и в бэкенд-отбор фото (§5) как приоритет. «Как
часто ходит» и «точная сумма депозитов» сегодня аппроксимируются через `balance` +
`registration_date`; если понадобится явный `deposits_total` / частота визитов — это
**сознательное расширение** `_CONTEXT_FIELDS` (+ handshake + тест-профиль) отдельным шагом,
не блокирует старт.

> **Фундамент (обязателен сразу).** При обмене nonce сохраняем связку
> **`tg_user_id ↔ player_id`** + снимок профиля в `retention_users`
> (+ `profile_source, profile_updated_at`). Именно связка делает последующую синхронизацию
> возможной — её нельзя отложить. Последний handshake перезаписывает профиль (last-wins).

### Свежесть профиля — три уровня, деградируют мягко

1. **Снапшот + ре-handshake** *(старт)* — профиль = значения с последнего входа; каждый новый диплинк их обновляет. Ноль интеграции у партнёра.
2. **Ленивый pull** *(дефолт при наличии API)* — на продукте `player_api_url` + ключ (зашифрован). На старте сессии (или перед гейтнутым фото), если снапшоту > TTL (напр. 1 ч), дёргаем наш эндпоинт → свежий профиль целиком, кэшируем.
3. **Push-webhook** *(апгрейд)* — CRM казино при изменении шлёт `POST /partner/{product}/player-update` (авторизация partner-секретом). Мгновенно, без опроса.

Партнёр без интеграции живёт на снапшоте — схема **деградирует, а не ломается**.

### Player API — контракт на нашей стороне

Форму задаём **мы** (как удобно нам), отдаём разработчикам казино как готовый контракт —
короткие правки от них учтём. Черновой контракт:

```
# pull (уровень 2): казино реализует у себя эндпоинт по нашей спеке
GET  {player_api_url}?player_id=<id>
Authorization: Bearer <player_api_key>
   → 200 { id, full_name, email, activation_status,
          country, balance, vip_level, registration_date }   # = _CONTEXT_FIELDS

# push (уровень 3): казино шлёт нам при изменении профиля
POST /partner/{product}/player-update
Authorization: Bearer <partner_secret>
     { player_id, ...любые изменившиеся поля профиля }
```

Поля — тот же whitelisted-набор, что в handshake. Отсутствующие поля не трогаем (частичный апдейт).

## 9. Мультитенантность

Всё ретеншн-хозяйство вешаем **на продукт** — white-label работает из коробки, как виджет.
Продукт резолвится по токену бота (аналог `widget_key → product`).

| На строке продукта | Секрет? |
|---|---|
| `telegram_bot_token_enc`, `telegram_bot_username` | токен — да (`_enc`/secretbox) |
| `telegram_channel_id`, `telegram_channel_url` | нет |
| `player_api_url`, `player_api_key_enc` (профиль-pull) | ключ — да (`_enc`) |
| `retention_enabled` | нет |
| Пул менеджеров, медиатека, retention-KB, retention prompt-variables | per-product стораджи |

Секреты — через `PUT /admin/products/{id}/secrets` (наружу только `has_*` флаги), как
OpenAI-ключи. Авторизация новых admin-роутов — строго через `require_admin` + scope-хелперы
против конкретного продукта. Подключение партнёра: завёл бота у BotFather → вбил
токен+канал в секреты → залил фото+менеджеров+retention-KB → перевёл persona-vars. Без
форков кода.

## 10. Модель данных

Правила БД без изменений: схема в `db.init_db()`, новые колонки через guarded
`ALTER … IF NOT EXISTS` (в `_ensure_columns()`), доступ только через `db.*` хелперы, каждая
per-turn/per-session строка несёт `product_id`. KB-контент поддержки лежит в `kb_topics` +
`kb_entries`; retention-KB — отдельный стор, их не трогает.

> **Сверено с кодовой базой ✓** — Профиль = `prompts._CONTEXT_FIELDS` (точное совпадение).
> Новые сентинелы не конфликтуют с существующими. `retention` добавляется к `SETTING_KEYS`.
> Секреты — суффикс `_enc` + secretbox (как `openai_key_*_enc`). Канал = существующий
> `consumer` (не новый `channel`); режим выводится из него. `player_id`/`conv_lang`/
> `context_reset_id` уже на `chat_sessions`. Telegram-кода в репозитории ещё нет.

### Новые таблицы

| Таблица | Ключевые поля | Назначение |
|---|---|---|
| `retention_users` | product_id, tg_user_id, tg_username, player_id, entry_type, **vip_level, balance, country, activation_status, registration_date, full_name** (снимок `_CONTEXT_FIELDS`), profile_source, profile_updated_at, unlocked_stage, last_stage_advance_at, assigned_manager_id, subscribed, photos_day, photos_sent_today, conv_lang, last_active_at | игрок в боте + связка + снимок профиля + состояние прогрессии/лимитов |
| `retention_kb` | product_id, title, trigger_when, body, links[], sort_order, active, updated_by, updated_at | единая retention-KB: сценарии/офферы/линки (грузится целиком в Layer 2) |
| `retention_photos` | product_id, storage_ref, telegram_file_id, description, tags[], level_min, stage, sort_order, category, active, views_count, created_by | каталог фото бренда |
| `retention_photo_views` | photo_id, retention_user_id, product_id, viewed_at | кто что видел (анти-повтор, счётчики) |
| `retention_managers` | product_id, display_name, username, active, assigned_count | пул менеджеров + round-robin |
| `retention_nonces` | nonce, product_id, payload, escalation, used, expires_at | одноразовый обмен диплинка |

### Правки существующих

- `products`: `+ telegram_bot_token_enc`, `player_api_key_enc` (оба через secretbox — как
  `openai_key_*_enc` / `handshake_secret_enc`) + не-секретные `telegram_bot_username`,
  `telegram_channel_id`, `telegram_channel_url`, `player_api_url`, `retention_enabled`.
- `chat_sessions`: **переиспользуем существующий `consumer`** (`'web'` → `'telegram'`) как
  канал — отдельный `channel` НЕ добавляем. Режим промпта **выводится из `consumer`**
  (telegram ⇒ retention; поддержку в TG не дублируем, решение 8), отдельный `mode` не нужен.
  `player_id`, `conv_lang`, `context_reset_id` уже есть. Добавляем только `+ tg_user_id`
  (nullable; связка живёт в `retention_users`). Переписка Ники персистится как обычные ходы
  (`chat_messages` + `ai_interaction_logs`, атомарно).

## 11. Эндпоинты и сентинелы

### Новые роуты

| Метод / путь | Кто зовёт | Делает |
|---|---|---|
| `POST /telegram/webhook/{secret}` | Telegram | апдейты бота; резолв продукта по токену; `/start`, сообщения, callback-кнопки |
| `POST /api/retention/deeplink` | сайт/виджет | handshake → `{nonce, deep_link}` |
| `POST /partner/{product}/player-update` | CRM казино | push профиля игрока, частичный (авторизация partner-секретом) |
| `GET/POST/PUT/DELETE /admin/retention/kb` | админ | CRUD единой retention-KB (сценарии + линки) |
| `GET/POST/PUT/DELETE /admin/retention/photos` + upload | админ | CRUD медиатеки + загрузка на Volume |
| `GET/PUT /admin/retention/managers` | админ | пул менеджеров |
| `GET/PUT /admin/retention/config` | админ | группа `retention` + persona-vars |
| `GET /admin/retention/users` | админ | аналитика вовлечённости / просмотров |

### Сентинелы retention-режима

| Сентинел | Что значит | Обработка бэкенда |
|---|---|---|
| `[[PHOTO:id]]` | отправить фото id из списка кандидатов | валидация id по допустимому набору → отправка → view → грунтинг |
| `[[STAGE_UP]]` | хинт «игрок готов к следующей ступени» | гард: порог + спейсинг + потолок тира; иначе игнор |
| `[[HANDOFF]]` | всплыла поддержка/жалоба/блок/ввод-вывод/RG/просьба о человеке → роутим наружу | эскалир. вход: кнопка менеджера; ретеншн-вход: возврат в поддержку сайта. Пишет `admin_events('retention_handoff', {reason, target})` |
| `[[LANG:xx]]` | язык ответа (как сейчас) | strip + стики `conv_lang` |

Все сентинелы стрипаются перед показом, как существующие. Топик-роутинг / suggest / resolved
/ KB-grounding в ретеншн-режиме **не используются** — это support-механики, а Ника
поддержкой не занимается (§1, §3).

## 12. Админка: три раздела

Админку делим на **три верхнеуровневых раздела**, чтобы настройки поддержки и ретеншна не
путались. Product-switcher, авторизация и роли — общие для всех трёх.

- **Раздел A · Поддержка** — всё про support-бота на сайте. База знаний (топики + тексты +
  переменные), Промпт (preview, prompt-variables, escalation-ключевые слова, тест-профиль),
  Переводы (виджет + сервисные реплики + contact_url), аналитика поддержки (по топикам,
  очередь Unresolved).
- **Раздел B · Retention / Telegram** — всё про ретеншн-бота и Telegram. **Retention KB /
  Линки** (единая база сценариев + офферов + линков — §4), Медиатека (фото: загрузка,
  описание, теги, `level_min`, `stage`, категория, просмотры), Менеджеры (пул + раздача),
  Ретеншн-персона (retention prompt-variables), Telegram-конфиг (bot token, channel, player
  API — через секреты продукта), группа настроек `retention` (§6), аналитика ретеншна
  (вовлечённость, показы фото, переходы к менеджерам, `[[HANDOFF]]`).
- **Раздел C · Общее / инфраструктура** — общее, не относящееся к конкретному боту.
  Структура (партнёры/продукты, widget-ключи, секреты), Пользователи + membership'ы, общие
  настройки (`general`: TTL сессий/токенов, лимиты), а также **shared-группы**, влияющие на
  оба бота, — `model` (OpenAI) и `language` — и общий обзорный дашборд.

> **Куда лёг какой knob.** Группа `retention` (§6) → раздел B. Группы `model` и `language`
> — shared, живут в разделе C. Группа `escalation` (ключевые слова) и support-переменные →
> раздел A. Токен бота / player API key — секреты продукта, доступны из раздела B, пишутся
> через общий `PUT /admin/products/{id}/secrets`.

## 13. Инфраструктура (Railway)

**Один сервис**, не новый. Telegram — это webhook-потребитель того же FastAPI + Postgres +
AI-ядра.

- Добавляем роут `POST /telegram/webhook` (верификация через Telegram `secret_token`
  header), модуль `telegram_transport` (парсинг апдейтов, рендер, inline-кнопки, отправка
  фото), модуль `retention` (режим-промпт + выбор фото), обмен nonce, роутинг менеджеров.
- **Object storage = Railway Volume** — бинарники фото на волюме сервиса. В Postgres —
  только метаданные; первый показ грузит с волюма, дальше `telegram_file_id` (повторная
  загрузка не нужна). Учесть бэкап волюма при деплое.
- Общий Postgres, общая админка. Выделять отдельный сервис — только если упрёшься в
  нагрузку; граница транспорт/ядро оставлена чистой, чтобы вынести можно было позже.

## 14. Безопасность и комплаенс

- **Nonce** одноразовый, TTL ~120 c, помечается `used` при обмене — диплинк нельзя переслать/переиспользовать.
- **Секреты** (bot token, player API key) — зашифрованы через `secretbox`, наружу только `has_*`.
- **Webhook** Telegram защищён `secret_token`; partner-webhook (профиль push) — partner-секретом.
- **18+ политика опущена на нашей стороне** — «шкала откровенности» и её границы задаются
  напрямую из админки продукта (уровни/ступени фото), отдельного гейта в коде не строим.
- **GDPR / срок хранения** — спец-механики хранения/удаления не строим; связка
  `tg_user_id ↔ player_id` + профиль хранятся бессрочно, как остальные данные сервиса.
- **Логирование передачи менеджеру** — пишем только *факт*:
  `admin_events('retention_manager_handoff', {tg_user_id, manager_id, from_reason})`.
  Дальнейший личный диалог в Telegram нам недоступен и не логируется.
- Авторизация admin-роутов — против конкретного продукта (не «админ где-то»).

## 15. План разработки

1. **Фундамент бота** — конфиг продукта (bot token, channel), webhook-скелет, обмен nonce с
   сайта, гейт подписки, связка `retention_users`, меню по типу входа. *Разблокирует
   остальное.*
2. **Ретеншн-Ника (текст) + retention-KB** — retention-сборка Layer-1 в `prompts.py`
   (`mode`), стор `retention_kb` + подстановка её целиком в Layer 2 (support-KB не грузится),
   admin-CRUD KB, режим в `chat_service`, рендер в Telegram, персист ходов. Живая переписка
   без фото.
3. **Медиатека** — таблицы, admin-загрузка, Volume + file_id-кэш, список кандидатов +
   `[[PHOTO]]` + грунтинг, гейтинг tier×stage×seen, дневной лимит, проактивность, прогрессия
   + `[[STAGE_UP]]`.
4. **Менеджеры** — пул, round-robin, липкое назначение, кнопка передачи, логирование.
5. **Профиль-синк** — снапшот (готов с ф.1) → ленивый pull по нашему Player API (когда
   казино реализует) → push-webhook. Таргетинг фото/тона по полному профилю.
6. **Аналитика и полировка** — дашборд вовлечённости/показов/переходов, стоимость, экспорт.

## 16. Решения и открытые вопросы

### Зафиксировано

- Ника — **только ретеншн.** Поддержку/жалобы/блок/ввод-вывод/RG/просьбу о человеке не обрабатывает, роутит наружу через `[[HANDOFF]]` (§3). Комбинированного бота не строим.
- **Отдельная retention-KB.** Support-KB (7 разделов) в ретеншн НЕ грузится; retention использует свою единую плоскую базу сценариев/линков в Layer 2 (§4). Ядро одно — промпт и KB по режимам разные.
- **Таргетинг по всему профилю**, не только VIP (§8).
- **Админка — 3 раздела**: Поддержка / Retention·Telegram / Общее (§12).
- **Player API — контракт задаём мы** (§8), отдаём разработчикам казино; их короткие правки учтём.
- **18+ политика опущена** — границы задаются уровнями/ступенями фото из админки продукта.
- **Object storage = Railway Volume** (§13).
- **Логирование передачи менеджеру = только факт** (§14). «RG» = responsible gaming; это поведение теперь часть единого правила роутинга наружу.
- **GDPR / срок хранения — ничего не делаем.**
- **Поля профиля = существующий тест-профиль** (`_CONTEXT_FIELDS`, §8).
- **Вход — только с сайта, всегда с данными**; органики нет; в бота из канала не пускаем.
- **Инфраструктура — один сервис** Railway, webhook поверх текущего FastAPI.

### Открытых блокеров нет

Все ключевые решения приняты и сверены с кодовой базой. Порядок работ — §15, фазы 1→6.
Из необязательного на будущее: при желании — явные поля `deposits_total` / частота визитов
(расширение `_CONTEXT_FIELDS`), выборка релевантных сценариев retention-KB (если разрастётся),
Telegram Mini App как апгрейд UI.
