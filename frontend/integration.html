Support Chat — интеграция и API

Документация для команд разработки: как подключить AI-чат поддержки к сайту казино (CMS), как передавать данные игрока, и какие API доступны для интеграции с основным бэкендом/фронтендом. База URL во всех примерах — адрес этого сервиса (страница открыта с него же).

1. Архитектура и мультитенантность

Сервис — standalone FastAPI-микросервис (HTTP/JSON), мультитенантный:

  • Партнёр — клиент платформы; владеет несколькими продуктами.
  • Продукт — одно казино. У каждого продукта свои: база знаний (топики + тексты + переменные), промпт-переменные (имя персонажа, бренд, tone of voice), переводы всех видимых игроку строк, настройки (модель, антиспам, языки, лимиты), свои OpenAI-ключи (1–2, с failover) и свой секрет handshake. Всё редактируется в админке этого сервиса.
  • Продукт идентифицируется публичным widget key (wk_…) — выдаётся в админке (Structure), передаётся виджетом при создании сессии. Ключ не секрет (лежит в HTML страницы), но неугадываем и его можно ротировать.
  • Секреты продукта (OpenAI-ключи, handshake-секрет) хранятся в БД в зашифрованном виде (мастер-ключ — только в окружении деплоя) и через API никогда не возвращаются — только флаги «задан/не задан». Если у продукта нет своих ключей, используется глобальный ключ деплоя.
Сессия чата привязана к продукту. Все ответы, копирайт, эскалация и статистика считаются в разрезе продукта; админка агрегирует по продукту, по партнёру и глобально.

2. Подключение виджета на сайт (CMS)

Один сниппет на странице казино. Widget key берётся из админки: Structure → продукт → Widget key (там же есть кнопка «Copy embed snippet»).

<link rel="stylesheet" href="https://CHAT_HOST/widget.css">
<script type="module" src="https://CHAT_HOST/widget.js"
        data-widget-key="wk_XXXXXXXXXXXX"></script>

Альтернативно — программный монтаж с переопределениями:

import { mount } from "https://CHAT_HOST/widget.js";
mount({
  API_BASE: "https://CHAT_HOST",
  WIDGET_KEY: "wk_XXXXXXXXXXXX",
  SIGNED_CONTEXT: "<подписанный blob с данными игрока — см. раздел 3>",
});
  • Виджет сам резолвит язык интерфейса из браузера, тянет топики и переводы своего продукта (GET /api/chat/topics, GET /api/chat/i18n) и создаёт сессию при выборе темы.
  • Ключ также можно задать глобально: window.NPCHAT_WIDGET_KEY = "wk_…" до загрузки скрипта.
  • Без ключа виджет попадает в «default» продукт деплоя — режим одиночного продукта/дев-стенда.
  • Не забудьте добавить домен сайта в CORS_ALLOW_ORIGINS сервиса.
Ротация widget key немедленно отключает старые сниппеты — обновляйте встраивание на сайте синхронно с ротацией.

3. Передача данных игрока — подписанный handshake (push)

Чтобы модель знала имя/статус/баланс игрока, бэкенд CMS подписывает данные игрока HMAC-секретом продукта и передаёт blob виджету (например, вставляет в страницу при рендере). Виджет отправляет его в POST /api/chat/session как signed_context. Неподписанные данные из браузера в продакшн-режиме игнорируются — доверяем только подписи бэкенда.

Формат blob: base64url(payload_json) + "." + base64url(HMAC_SHA256(secret, base64url(payload_json)))

Payload — JSON c полями игрока + iat/exp (unix-секунды; TTL по умолчанию ≤ 300 сек, anti-replay). Модель видит только whitelisted-поля:

{
  "id": "player-123",
  "full_name": "Иван Петров",
  "email": "ivan@example.com",
  "activation_status": "active",
  "country": "Portugal",
  "balance": "1500.00 EUR",
  "vip_level": "Silver",
  "registration_date": "2024-01-15",
  "iat": 1751500000,
  "exp": 1751500300
}

Пример подписи (Node.js):

const crypto = require("crypto");
const b64url = (buf) => Buffer.from(buf).toString("base64url");

function signHandshake(context, secret, ttlSec = 300) {
  const now = Math.floor(Date.now() / 1000);
  const payload = JSON.stringify(
    Object.fromEntries(Object.entries({ ...context, iat: now, exp: now + ttlSec }).sort()),
  );
  const p64 = b64url(payload);
  const sig = crypto.createHmac("sha256", secret).update(p64).digest("base64url");
  return `${p64}.${sig}`;
}

Пример подписи (Python):

import base64, hashlib, hmac, json, time

def sign_handshake(context: dict, secret: str, ttl_sec: int = 300) -> str:
    now = int(time.time())
    payload = {**context, "iat": now, "exp": now + ttl_sec}
    p64 = base64.urlsafe_b64encode(
        json.dumps(payload, separators=(",", ":"), sort_keys=True).encode()
    ).rstrip(b"=").decode()
    sig = hmac.new(secret.encode(), p64.encode(), hashlib.sha256).digest()
    s64 = base64.urlsafe_b64encode(sig).rstrip(b"=").decode()
    return f"{p64}.{s64}"
  • Секрет handshake задаётся per product в админке (Structure → Product secrets) — у каждого казино свой.
  • Blob живёт максимум 300 секунд — генерируйте его при рендере страницы или отдавайте отдельным endpoint'ом своего бэкенда непосредственно перед открытием чата.
  • Pull-интеграция (сервис сам ходит в API CMS за данными игрока) — запланированный этап; контракт будет добавлен сюда, когда зафиксируем API CMS.

4. Публичный Chat API (используется виджетом)

Все запросы/ответы — JSON. Сессия = UUID + bearer-токен, который возвращает POST /session. Контракт consumer-agnostic: свой фронтенд может ходить в эти же ручки напрямую — но тогда он обязан воспроизвести клиентскую логику виджета, см. раздел 5.

POST/api/chat/session

Создать сессию. Тело: { widget_key?, locale?, signed_context?, consumer?, recaptcha_token? }. Ответ: { session_id, token, topics[], lang, languages[] }. Ошибки: 403 bad_widget_key, 401 bad_handshake, 403 recaptcha_failed, 429.

GET/api/chat/topics?widget_key=&lang=

Каталог тем продукта без сессии (для мгновенной первой отрисовки). Кэшируется 60 сек.

GET/api/chat/i18n?widget_key=

Все строки виджета по языкам продукта (правки из админки доезжают без редеплоя виджета).

POST/api/chat/topic

{ session_id, topic_slug } + Authorization: Bearer <token>. Выбор/смена темы.

POST/api/chat/message

{ session_id, text, closing? } + bearer. Один ход диалога. Ответ:

{
  "reply": "текст ответа (может быть пуст при эскалации/переключении темы)",
  "lang": "ru",
  "escalation": { "active": false } | {
      "active": true, "final": true|false, "retention": true?,
      "message": "...", "button": { "label": "...", "url": "..." } },
  "message_count": 5,
  "suggested_topic": null | { "slug": "...", "title": "..." },
  "suggestions": ["наводящий вопрос 1?", "вопрос 2?"],
  "closing_suggestion": null | "Проблема решена.",
  "resolved": false
}

Ошибки: 401, 409 session_closed (сессия закрыта — начать новую), 429 (rate-limit/cooldown), 400 (слишком длинно / injection), 413/411 (body cap). escalation.final=true — диалог завершён (hand-off), final=false — карточка показана, чат продолжается.

GET/api/chat/session/{id}

Резюме сессии: статус + история (bearer обязателен).

POST/api/chat/escalate · POST/api/chat/resolve

Явная эскалация (возвращает карточку контакта, закрывает сессию) / завершение чата игроком.

5. Свой UI поверх Chat API — обязательная клиентская логика

Готовый виджет (widget.js) — это не только вёрстка: в нём заложена клиентская логика, без которой чат работает медленно, «зависает» или ломает контракт с промптом. Если вы строите собственный интерфейс напрямую поверх Chat API (раздел 4), всё перечисленное ниже — часть контракта, которую ваш клиент обязан воспроизвести. Раздел написан по фактическому поведению виджета; сверяйтесь с widget.js как с референс-реализацией.

5.1. Старт и первая отрисовка — скорость

  • Язык интерфейса резолвится на клиенте, до любого запроса — из navigator.languages[0] || navigator.language (фолбэк — английский). Не ждите ответа сервера, чтобы выбрать язык шелла: иначе чат «открывается на английском и через секунды перепрыгивает на русский».
  • Прогрейте данные при монтировании страницы, а не при открытии панели: GET /api/chat/topics и GET /api/chat/i18n — оба работают без сессии и кэшируются (60 сек на сервере). Тогда список тем рисуется мгновенно.
  • Строки интерфейса: держите запечённую копию базовых строк для мгновенной первой отрисовки и накатывайте поверх ответ GET /i18n — так правки копирайта из админки доезжают до вашего UI без редеплоя фронта.
  • Скрипт reCaptcha подгружайте заранее (при монтировании), а не в момент создания сессии — это самая медленная сторонняя загрузка на пути «тап по теме → чат готов».

5.2. Жизненный цикл сессии — экономия и корректность

  • Сессию создавайте лениво — только когда игрок реально выбрал тему. Не создавайте её при открытии панели: каждый «открыл-закрыл» посетитель иначе жжёт per-IP лимит на POST /session и плодит пустые сессии в статистике.
  • Тап по теме — оптимистичный. Покажите экран диалога сразу (он не требует сети), а медленную цепочку — reCaptcha-токен → POST /session → POST /topic — запустите в фоне и сохраните её promise. Первая отправка сообщения просто ждёт этот promise (обычно он уже завершён, пока игрок печатает). Если цепочка упала — верните игрока к списку тем с локализованной ошибкой. Неудавшийся POST /topic не фатален (чат работает и без темы), неудавшийся POST /session — фатален.
  • Не кэшируйте отклонённый promise создания сессии: временный сбой (сеть, 429, reCaptcha) не должен намертво заклинить все последующие попытки — сбросьте promise при ошибке, чтобы следующий тап повторил создание заново.
  • Защита от двойного тапа по кнопке темы: выставляйте флаг синхронно до любого await, иначе двойной клик создаст два диалога/двойное приветствие.
  • Каждый вход — новый разговор. «Назад», закрытие панели, завершение чата и эскалация сбрасывают локальные session_id/token; повторное открытие всегда начинает с чистого списка тем и новой сессии. Не резюмируйте старую сессию на клиенте — она может быть уже закрыта на сервере, и первое сообщение «нового» чата молча упрётся в 409.
  • Токен сессии держите в памяти, не в localStorage: он живёт один разговор и не должен переживать вкладку.

5.3. Канонное приветствие — контракт с промптом

Это самый неочевидный пункт. Пузырь «Привет, я Ника, чем могу тебе помочь?» рисует клиент (строка greeting из /i18n) сразу при выборе темы, до первого сообщения игрока; на сервер он не отправляется и в истории не хранится. Промпт модели построен в расчёте на то, что это приветствие уже показано: модель никогда не представляется, а своё единственное приветствие («Привет, Андрей!») даёт по имени в первом ответе — только если имя игрока передано (handshake, раздел 3). Если ваш UI не рисует канонный пузырь, диалог начнётся вообще без «здравствуйте» — добавьте его на клиенте, серверной альтернативы нет.

5.4. Обработка ответа POST /message — side-payload'ы

Один ход возвращает не только текст (поля — в разделе 4). Клиент обязан обработать каждое:

  • reply может быть пустым — при эскалации и при переключении темы. Никогда не рисуйте пустой пузырь: если текст пуст, снимите индикатор «печатает» и работайте только с side-payload'ами.
  • escalation.active=true, final=true (жёсткая эскалация): покажите карточку (message + кнопка button.url, если url задан; открывать с target="_blank" rel="noopener"), скройте поле ввода и сбросьте локальную сессию — сервер отклонит дальнейшие ходы с 409. final=false (мягкая, по ключевым словам): карточку покажите, но поле ввода оставьте — разговор продолжается. escalation.retention=true — этот хендофф ведёт не на форму контакта, а в Telegram-бота Nika: button.url уже содержит одноразовый диплинк входа (nonce живёт ~120 c), просто откройте его — игрок попадёт в бота, подпишется на канал и получит кнопку менеджера. Флаг появляется только когда у продукта включён ретеншн; иначе кнопка ведёт на статический contact_url.
  • suggested_topic — автопереключение темы. Сервер уже подавил необоснованный ответ (он генерировался без базы знаний целевой темы), поэтому reply пуст. Клиент: (1) оставляет в переписке постоянную заметку «переключаю на «X»…» (без кнопок — это запись о переходе); (2) вызывает POST /topic с новым slug; (3) после короткой паузы ~900 мс (чтобы переход читался, а не мелькал) повторно отправляет исходный текст игрока в POST /message — этот второй ответ и есть то, что видит игрок. Ограничьте цепочку (виджет: максимум 2 автопереключения подряд): если лимит выбран, покажите локализованный фолбэк «не удалось определить тему, переформулируйте» — иначе ход закончится вообще без ответа и чат будет выглядеть зависшим.
  • suggestions[] (до 2 наводящих вопросов): кнопки-пузыри у поля ввода; тап отправляет текст вопроса как обычный ход. Старые пузыри убирайте в момент начала любого нового хода — они устаревают мгновенно.
  • closing_suggestion («Проблема решена.»): отдельно стилизованный завершающий пузырь. Тап: отправить его текст с closing=true (модель ответит коротким прощанием), затем best-effort POST /resolve и завершить разговор. После этого не показывать кнопку «завершить чат» — игрок уже завершил. Если этот ход неожиданно вернул эскалацию — обработать её как обычно вместо закрытия.
  • resolved=true: показать кнопку «завершить чат»; тап → POST /resolve (best-effort — интерфейс закрывается в любом случае) + свернуть чат. Одновременно показывается только один финишный контрол: при конфликте resolved побеждает closing_suggestion.
  • lang — дрейф языка разговора. Если игрок начал писать на другом поддерживаемом языке, ответы переключаются на него; переключайте вслед и хром (кнопки, плейсхолдер, канонное приветствие, заголовки тем — GET /topics уже на новом языке), и передавайте новый язык как locale при создании следующей сессии — иначе после «назад» язык отскочит к языку браузера.

5.5. Ошибки и деградация

  • 409 session_closed: не показывать сырую ошибку — молча завершить разговор локально (следующее обращение начнёт новую сессию).
  • 429 (rate-limit / cooldown между сообщениями): локализованная заметка «слишком часто», без автоматических ретраев в цикле.
  • 400 (слишком длинное сообщение / injection-фильтр): показать локализованный текст ошибки.
  • Сбой модели не приходит ошибкой: при недоступности OpenAI сервер возвращает обычный 200 с локализованной просьбой отправить сообщение ещё раз — для клиента это обычный ход, ничего специального делать не нужно.
  • Индикатор «печатает» (пузырь с точками) на время запроса; ответ заменяет его на месте. Сетевая ошибка — локализованный текст ошибки в том же пузыре. Не отправляйте новый ход, пока предыдущий в полёте.

5.6. Безопасность рендеринга

  • Ответы ассистента приходят с лёгким Markdown (модель этому обучена директивой): жирный/курсив, `код`, ссылки, короткие списки. Рендерите только этот сабсет и обязательно в порядке «сначала HTML-escape всего текста, потом восстановление разметки по белому списку»; ссылки — только http(s)/mailto, с rel="noopener". Никакой сырой HTML из модели не должен попадать в DOM. Таблицы/код-блоки модель не шлёт (директива), но неизвестную разметку показывайте как текст, не рендерьте.
  • Текст игрока рендерится строго литерально (textContent, не innerHTML) — и его собственный ввод, и его сообщения в истории.
  • Widget key — не секрет (он в HTML), но bearer-токен сессии — секрет одного разговора: только в память, только в заголовок Authorization.
  • reCaptcha v3: токен нужен при создании сессии, если на сервисе задан RECAPTCHA_SECRET; без него сервис пропускает проверку (dev). Домен вашего сайта должен быть в CORS_ALLOW_ORIGINS.

5.7. Мобильные нюансы

  • Определяйте «мобильность» по реальному window.visualViewport, а не по CSS media-query: media-query меряет CSS-пиксели, которыми управляет viewport-мета вашей страницы — при нестандартной мете полноэкранный режим просто не срабатывает.
  • При открытой экранной клавиатуре layout-viewport не меняется, а visual-viewport сжимается — панель с position:fixed на всю высоту прячет поле ввода под клавиатурой. Виджет решает это, подгоняя top/left/width/height панели под visualViewport на каждый его resize/scroll — повторите этот приём.
  • Держите ленту прикнопленной к последнему сообщению и повторяйте пиннинг после следующего кадра (rAF): пузыри-подсказки и рендер Markdown меняют высоту после вставки ответа, иначе свежий ответ оказывается наполовину скрыт.

Выпадение любого пункта — это «чат тормозит/завис/ молчит» в проде.

6. Admin API — интеграция с основным бэкендом/фронтендом

Все ручки под /admin/*, JSON, авторизация — bearer JWT из POST /admin/login. Именно этим API пользуется наша админка («хаб»); ваш бэкенд может использовать его же для встраивания управления и статистики в основную платформу — заведите отдельный сервисный аккаунт с нужным скоупом.

POST/admin/login

{ email, password } → { token, role, memberships[] }. Токен живёт admin_token_ttl_min (по умолчанию 480 мин); аккаунт и права перепроверяются на каждом запросе.

Скоупинг запросов

Почти все ручки принимают ?product_id=N (один продукт) или ?partner_id=N (агрегат по партнёру). Без параметров — всё, что доступно аккаунту. Запрос вне своего скоупа → 403.

Структура (партнёры/продукты)

РучкаЧто делает
GET/admin/structure Дерево партнёры → продукты, видимое аккаунту (+ widget key, флаги секретов).
POST/admin/partners {slug, name} — создать партнёра (глобальный админ).
PUT/admin/partners/{id} {name?, active?}.
POST/admin/products {partner_id, slug, name} — создать казино (widget key генерируется, KB-переменные сеются).
PUT/admin/products/{id} {name?, active?}.
PUT/admin/products/{id}/secrets {openai_key_primary?, openai_key_fallback?, handshake_secret?} — write-only; пустая строка очищает поле; хранится шифрованно; в ответ только has_*-флаги.
POST/admin/products/{id}/widget-key Ротация widget key (старые сниппеты перестают работать сразу).

Контент продукта

РучкаЧто редактирует
/admin/kb/topics · /admin/kb/content · /admin/kb/variables Темы, KB-тексты, {placeholder}-переменные базы знаний.
/admin/prompt-variables Брендирование промпта: имя персонажа, бренд, продукты, tone of voice.

Пользователи и доступы

РучкаЧто делает
GET/admin/users Аккаунты в зоне доступа + их membership'ы.
POST/admin/users {email, password, role, scope_type: global|partner|product, partner_id?, product_id?}.
POST/admin/users/{email}/memberships Выдать доступ (скоуп + роль). Повторная выдача обновляет роль.
DELETE/admin/users/{email}/memberships/{id} Отозвать доступ.
PUT / DELETE /admin/users/{email} Пароль/активность / удаление аккаунта (в пределах своего скоупа).

7. Роли и доступы

Скоупadminmanager
global — весь хаб всё: партнёры, продукты, ключи, пользователи, глобальные настройки чтение всего
partner — партнёр и все его казино создание продуктов партнёра, их контент/настройки/ключи, пользователи в скоупе чтение по партнёру
product — одно казино контент/настройки/ключи этого казино, его пользователи чтение по казино

У аккаунта может быть несколько membership'ов (например, manager глобально + admin одного продукта). Сервер — единственный источник истины: каждый запрос проверяется по БД, отзыв доступа действует немедленно.

8. Переменные окружения (деплой)

ПеременнаяНазначение
DATABASE_URLPostgres (обязательно).
OPENAI_API_KEY / OPENAI_API_KEY_FALLBACK Глобальные ключи-фолбэк для продуктов без своих ключей.
SESSION_JWT_SECRET / ADMIN_JWT_SECRET Подпись сессионных и админских токенов (в проде — разные значения).
SECRETS_MASTER_KEY Мастер-ключ шифрования секретов продуктов в БД. Задать отдельным сильным значением; его смена делает сохранённые секреты нечитаемыми (перевводятся в админке).
WIDGET_HANDSHAKE_SECRET Деплой-фолбэк секрета handshake (у продукта приоритет — его собственный секрет).
CORS_ALLOW_ORIGINSДомены сайтов, где встраивается виджет.
RECAPTCHA_SECRET, TRUSTED_PROXY_*, CONTACT_FORM_URL… Прочее — см. README сервиса (полная таблица).
