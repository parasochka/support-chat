1. Страницы документации

Документация разбита по задачам — каждая страница самодостаточна и открывается прямо с сервиса:

СтраницаДля чего
/integration-widget Только виджет: сниппет встраивания на сайт (CMS), widget key, программный монтаж, Cloudflare Turnstile, CORS, чек-лист подключения.
/integration-data Передача и синхронизация данных игрока: whitelist полей, подписанный handshake (+ примеры кода), ленивый pull из Player API, push-webhook из CRM, сигналы активности, тест-профиль.
/integration-chat-api Публичный Chat API (справочник ручек) + обязательная клиентская логика, если вы строите собственный UI чата вместо готового виджета.
/integration-telegram Retention-бот в Telegram: диплинк и nonce, гейт подписки на канал, проактивный агент (реакции на события казино), справочник API, настройка бота.
/integration-admin Интеграция с общей админкой: модель ролей, JWT-вход, сервисные API-ключи sak_…, скоупинг, справочник /admin/* эндпоинтов, примеры curl.
Типовой путь подключения нового казино: создать продукт в админке → встроить виджет → настроить handshake с данными игрока → (опционально) подключить Telegram-бота и синхронизацию профиля.

2. Архитектура и мультитенантность

Сервис — standalone FastAPI-микросервис (HTTP/JSON), мультитенантный:

  • Партнёр — клиент платформы; владеет несколькими продуктами.
  • Продукт — одно казино. У каждого продукта свои: база знаний (топики + тексты + переменные), промпт-переменные (имя персонажа, бренд, tone of voice), переводы всех видимых игроку строк, настройки (модель, антиспам, языки, лимиты), свои OpenAI-ключи (1–2, с failover) и свой секрет handshake. Всё редактируется в админке этого сервиса.
  • Продукт идентифицируется публичным widget key (wk_…) — выдаётся в админке (Structure), передаётся виджетом при создании сессии. Ключ не секрет (лежит в HTML страницы), но неугадываем и его можно ротировать.
  • Секреты продукта (OpenAI-ключи, handshake-секрет, токен Telegram-бота, ключ Player API, секрет Turnstile) хранятся в БД в зашифрованном виде (мастер-ключ — только в окружении деплоя) и через API никогда не возвращаются — только флаги «задан/не задан». Если у продукта нет своих ключей, используется глобальный ключ деплоя.
Сессия чата привязана к продукту. Все ответы, копирайт, эскалация и статистика считаются в разрезе продукта; админка агрегирует по продукту, по партнёру и глобально.

3. Переменные окружения (деплой)

Общие переменные деплоя сервиса; специфичные для Telegram-бота — на странице retention-бота.

ПеременнаяНазначение
DATABASE_URLPostgres (обязательно).
OPENAI_API_KEY / OPENAI_API_KEY_FALLBACK Глобальные ключи-фолбэк для продуктов без своих ключей.
SESSION_JWT_SECRET / ADMIN_JWT_SECRET Подпись сессионных и админских токенов (в проде — разные значения).
SECRETS_MASTER_KEY Мастер-ключ шифрования секретов продуктов в БД. Задать отдельным сильным значением; его смена делает сохранённые секреты нечитаемыми (перевводятся в админке).
WIDGET_HANDSHAKE_SECRET Деплой-фолбэк секрета handshake (у продукта приоритет — его собственный секрет).
CORS_ALLOW_ORIGINSДомены сайтов, где встраивается виджет.
TURNSTILE_SITE_KEY/TURNSTILE_SECRET (фолбэк-пара Cloudflare Turnstile — у продукта приоритетна его собственная, см. страницу виджета), TRUSTED_PROXY_*, CONTACT_FORM_URL… Прочее — см. README сервиса (полная таблица).
