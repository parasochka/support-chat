Интеграция с общей админкой — Admin API

Документация для команд разработки: как «мастер»-админка платформы (или любой партнёрский бэкенд) интегрируется с /admin/* API этого сервиса — авторизация людей и систем, модель ролей, скоупинг по партнёрам/продуктам и справочник основных групп эндпоинтов. 1. Модель ролей — 2 роли × 3 скоупа = 6 уровней доступа

Каждый субъект доступа — человек-аккаунт или сервисный ключ — несёт роль в скоупе. Ролей две: admin (чтение + запись в пределах скоупа) и manager (только чтение). Скоупов три: весь хаб, один партнёр (все его казино) или один продукт (одно казино).

Скоупadminmanager
global — весь хаб всё: партнёры, продукты, секреты, пользователи, глобальные настройки чтение всего
partner — партнёр и все его казино создание продуктов партнёра, их контент/настройки/секреты, пользователи в скоупе чтение по партнёру
product — одно казино контент/настройки/секреты этого казино, его пользователи чтение по казино

Человеческим аккаунтам доступы выдаются membership'ами (админка → вкладка Users; у аккаунта может быть несколько, например manager глобально + admin одного продукта). Сервисный ключ несёт ровно одну роль в одном скоупе. Сервер — единственный источник истины: права перепроверяются по БД на каждом запросе, отзыв действует немедленно. Запрос вне своего скоупа → 403.

2. Вход для людей — JWT

POST/admin/login

{ email, password } → { token, role, memberships[] }. Токен — bearer JWT, живёт admin_token_ttl_min (по умолчанию 480 мин), передаётся в каждом запросе заголовком Authorization: Bearer <token>. Аккаунт, активность и роль перепроверяются по БД на каждом запросе — деактивация/понижение применяются сразу, не дожидаясь истечения токена. Логин rate-limit'ится; неудачные попытки логируются.

JWT — для интерактивных сессий людей (наша админка работает именно так). Для интеграции «система-к-системе» логин по email/паролю не нужен — используйте сервисные ключи (раздел 3): они не истекают и не требуют перевыпуска токена.

3. Сервисные API-ключи sak_… — машинный доступ

Сервисный ключ — это долгоживущий bearer-токен вида sak_… для внешних систем (мастер-админка платформы, партнёрский бэкенд). Используется точно так же, как человеческий JWT: Authorization: Bearer sak_… на любом /admin/* эндпоинте — все проверки скоупа применяются без изменений, как будто запрос сделал аккаунт с одним membership'ом.

  • Как выпустить: в админке сервиса — раздел System → API keys (или напрямую POST /admin/api-keys). Задаются имя, роль (admin — запись / manager — только чтение) и скоуп (global / partner / product).
  • Токен показывается ровно один раз — в ответе на создание. В БД хранится только SHA-256-хэш; восстановить потерянный ключ нельзя, только выпустить новый.
  • Управляют ключами только люди: выпуск/деактивация/удаление требуют человеческого admin-аккаунта — утёкший ключ не может наплодить новых ключей. Менеджеры ключей не видят.
  • Деактивация и удаление действуют немедленно — ключ перестаёт работать на следующем запросе.
  • Выдавайте ключу минимальный достаточный скоуп: партнёрской интеграции — ключ скоупа partner, витрине одного казино — скоуп product, и manager, если запись не нужна.
РучкаЧто делает
GET/admin/api-keys Список ключей в admin-зоне вызывающего (без плейнтекста токенов).
POST/admin/api-keys {name, role, scope_type, partner_id?, product_id?} → {key, token} — единственный ответ с плейнтекстом токена.
PUT/admin/api-keys/{id} {name?, active?} — переименовать / деактивировать-активировать.
DELETE/admin/api-keys/{id} Удалить ключ (немедленно).
Ключ sak_… — полноценная админская креденшелка своего скоупа. Храните его как секрет (vault/env бэкенда), не кладите во фронтенд и не логируйте; при малейшем подозрении на утечку — деактивируйте и перевыпустите.

4. Скоуп-параметры запросов

Почти все ручки принимают ?product_id=N (один продукт) или ?partner_id=N (агрегат по партнёру). Без параметров запрос покрывает всё, что доступно субъекту (у глобального ключа — весь хаб). Правила:

  • Аналитика (/admin/overview, /admin/retention/overview, …) без скоупа агрегирует по всей зоне доступа; с product_id/partner_id — по срезу.
  • Контентные ручки (KB, переводы, prompt-переменные, retention-конфиг) работают per product — передавайте product_id явно.
  • /admin/settings/{group} с ?product_id= редактирует оверрайд продукта; без — глобальные значения по умолчанию (только глобальный админ).
  • Диапазоны дат: ?from=YYYY-MM-DD&to=YYYY-MM-DD; date-only to включает весь указанный день.

6. Примеры curl

Логин человеком → JWT

curl -s -X POST https://CHAT_HOST/admin/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@example.com","password":"···"}'
# → { "token": "eyJhbGciOi…", "role": "admin", "memberships": [...] }

Запрос сервисным ключом (без логина)

curl -s "https://CHAT_HOST/admin/structure" \
  -H "Authorization: Bearer sak_XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"

KPI-обзор по одному продукту за период

curl -s "https://CHAT_HOST/admin/overview?from=2026-07-01&to=2026-07-07&product_id=42" \
  -H "Authorization: Bearer sak_XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"
# тот же вызов работает и с человеческим JWT вместо sak_-ключа

7. OpenAPI-схема и /docs

Swagger/ReDoc и /openapi.json описывают всю поверхность сервиса (включая /admin), поэтому в продакшн-деплое они выключены по умолчанию. Для дев/стейджа их можно включить переменной окружения EXPOSE_API_DOCS=1 — тогда доступны /docs, /redoc и /openapi.json. В прод не включайте.
