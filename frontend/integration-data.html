Данные игрока — передача и синхронизация

Все способы, которыми сервис узнаёт об игроке и держит его профиль свежим: подписанный handshake с сайта (push при открытии чата), снапшот профиля в Telegram-боте, ленивый pull из Player API казино и push-webhook из CRM. Виджет — на странице виджета, Telegram-бот — на странице retention-бота. База URL во всех примерах — адрес этого сервиса.

1. Поля профиля — единый whitelist

Во всех контрактах ниже используется один и тот же набор полей. Модель видит только их — любые другие поля отбрасываются на входе:

ПолеПримерЗачем модели
id"player-123"идентификатор игрока (ключ pull/push)
full_name"Иван Петров"приветствие по имени
email"ivan@example.com"контекст аккаунта
activation_status"active"ответы про верификацию/блокировки
country"Portugal"региональный контекст
balance"1500.00 EUR"вопросы про баланс/вывод
vip_level"Silver"VIP-обращение; в ретеншне — гейты фото
registration_date"2024-01-15"стаж игрока
Поля с маркерами prompt-injection зануляются санитайзером до попадания в промпт. Никогда не передавайте в профиле номера карт, CVV, пароли, 2FA-коды — сервис их не запрашивает и не должен видеть.

2. Подписанный handshake (push с сайта)

Чтобы модель знала имя/статус/баланс игрока, бэкенд CMS подписывает данные игрока HMAC-секретом продукта и передаёт blob клиенту (например, вставляет в страницу при рендере). Неподписанные данные из браузера в продакшн-режиме игнорируются — доверяем только подписи бэкенда.

Формат blob: base64url(payload_json) + "." + base64url(HMAC_SHA256(secret, base64url(payload_json)))

Payload — JSON c полями игрока (раздел 1) + iat/exp (unix-секунды; TTL по умолчанию ≤ 300 сек, anti-replay):

{
  "id": "player-123",
  "full_name": "Иван Петров",
  "email": "ivan@example.com",
  "activation_status": "active",
  "country": "Portugal",
  "balance": "1500.00 EUR",
  "vip_level": "Silver",
  "registration_date": "2024-01-15",
  "iat": 1751500000,
  "exp": 1751500300
}

Пример подписи (Node.js):

const crypto = require("crypto");
const b64url = (buf) => Buffer.from(buf).toString("base64url");

function signHandshake(context, secret, ttlSec = 300) {
  const now = Math.floor(Date.now() / 1000);
  const payload = JSON.stringify(
    Object.fromEntries(Object.entries({ ...context, iat: now, exp: now + ttlSec }).sort()),
  );
  const p64 = b64url(payload);
  const sig = crypto.createHmac("sha256", secret).update(p64).digest("base64url");
  return `${p64}.${sig}`;
}

Пример подписи (Python):

import base64, hashlib, hmac, json, time

def sign_handshake(context: dict, secret: str, ttl_sec: int = 300) -> str:
    now = int(time.time())
    payload = {**context, "iat": now, "exp": now + ttl_sec}
    p64 = base64.urlsafe_b64encode(
        json.dumps(payload, separators=(",", ":"), sort_keys=True).encode()
    ).rstrip(b"=").decode()
    sig = hmac.new(secret.encode(), p64.encode(), hashlib.sha256).digest()
    s64 = base64.urlsafe_b64encode(sig).rstrip(b"=").decode()
    return f"{p64}.{s64}"
  • Секрет handshake задаётся per product в админке (Structure → Product secrets) — у каждого казино свой. Деплой-переменная WIDGET_HANDSHAKE_SECRET — только фолбэк.
  • Blob живёт максимум 300 секунд — генерируйте его при рендере страницы или отдавайте отдельным endpoint'ом своего бэкенда непосредственно перед открытием чата.

3. Куда передаётся handshake

КудаКакЧто происходит
Чат поддержки (виджет) поле signed_context в POST/api/chat/session (готовый виджет — через SIGNED_CONTEXT в mount(), см. страницу виджета) профиль виден модели всю сессию чата (персонализация, ответы про баланс/статус)
Telegram-бот (ретеншн) поле signed_context в POST/api/retention/deeplink (см. диплинк и nonce) при редиме /start фиксируется связка tg_user_id ↔ player_id и снапшот профиля; каждый новый диплинк обновляет снапшот

Дальше профиль в Telegram-боте живёт на снапшоте и деградирует мягко: без интеграции остаётся последний снапшот, а с Player API (pull) и/или CRM-webhook (push) — освежается автоматически.

4. Ленивый pull — Player API на стороне казино

Если на продукте заданы Player API URL + ключ (админка → Retention · Telegram → Telegram config + Secrets), сервис сам подтягивает свежий профиль перед ходом бота, когда снапшоту больше profile_pull_ttl_sec (по умолчанию 1 ч). Эндпоинт реализуете вы у себя:

GET  {player_api_url}?player_id=<id>
Authorization: Bearer <player_api_key>
→ 200 {
  "id": "player-123", "full_name": "Иван Петров", "email": "ivan@example.com",
  "activation_status": "active", "country": "Portugal", "balance": "1500.00 EUR",
  "vip_level": "Silver", "registration_date": "2024-01-15",
  "last_login_at": "2026-07-05T18:20:00Z",     // опционально — сигналы активности,
  "last_played_at": "2026-07-05T19:02:00Z",    // см. раздел 6
  "last_deposit_at": "2026-06-28T11:45:00Z"
}

Pull — best-effort: сбой запроса не ломает ход, снапшот просто остаётся прежним. Ключ Player API хранится у нас шифрованно (наружу — только флаг «задан»).

5. Push-webhook — CRM казино шлёт изменения мгновенно

POST/partner/{product_id}/player-update
POST /partner/42/player-update
Authorization: Bearer <handshake-секрет продукта>
{
  "player_id": "player-123",
  "balance": "2000.00 EUR",
  "vip_level": "Gold",
  "last_login_at": "2026-07-05T18:20:00Z",
  "last_played_at": "2026-07-05T19:02:00Z",
  "last_deposit_at": "2026-06-28T11:45:00Z"
}
→ 200 { "ok": true, "updated": 1 }

Авторизация push — секретом handshake продукта (тот же общий секрет, что подписывает handshake, раздел 2). Апдейт частичный: отсутствующие поля не трогаются.

6. Сигналы активности на стороне казино

Помимо профильных полей, pull- и push-контракты принимают опциональные ISO-8601-таймстемпы:

ПолеЧто означает
last_login_atпоследний вход на сайт казино
last_played_atпоследняя игра/ставка
last_deposit_atпоследний депозит
На эти таймстемпы опирается резолвер состояния игрока проактивного агента (страница retention-бота, раздел «Проактивный агент»): статус active/at-risk/dormant, дни с последнего депозита и т.п. Передавать можно любым из двух путей — push'ем (раздел 5) или в ответе своего Player API (раздел 4). Непарсибельные значения отбрасываются.

7. Канонические события — фид проактивного агента

POST/partner/{product_id}/event

Событийный webhook: казино шлёт значимые действия игрока по мере их наступления. Одно событие — плоским объектом, пачка — полем events (до 500 за запрос). Авторизация — тем же партнёрским секретом, что и push-webhook (раздел 5).

POST /partner/42/event
Authorization: Bearer <handshake-секрет продукта>
{
  "event_id":   "evt_01H8Y7Z9A0B1C2D3E4F5G6H7",   // ваш ключ идемпотентности (ULID/uuid)
  "event_name": "deposit_confirmed",               // каноническое имя, см. список ниже
  "player_id":  "player-123",
  "timestamp":  "2026-07-05T19:02:00Z",            // ISO-8601; когда произошло у вас
  "payload":    { "amount": 50.0, "currency": "USDT", "method": "trc20" },
  "tg_user_id": 123456789                          // опционально: точный Telegram-получатель
}
→ 200 { "ok": true, "stored": true, "duplicate": false }

// batch:
{ "events": [ { … }, { … } ] }
→ 200 { "ok": true, "stored": 2, "duplicates": 0, "errors": [] }

Канонические имена (события вне списка отклоняются с 422): session_started, session_ended, deposit_initiated, deposit_confirmed, deposit_failed, withdrawal_settled, bet_settled (payload: amount, win_amount, bonus_money), bonus_granted, bonus_claimed, bonus_completed, bonus_expired, kyc_started, kyc_approved, kyc_rejected, xp_granted, level_up, class_up, downgrade, highlights_pack_opened, highlights_pack_completed, check_in_done, mission_completed.

Получатель в Telegram по умолчанию резолвится по player_id — берётся его последняя активная привязка к боту. Если один player_id привязан к нескольким Telegram-аккаунтам (типично на тестовых стендах), передайте опциональный tg_user_id — реакция уйдёт ровно этому аккаунту; незнакомый tg_user_id не «падает» на другой аккаунт, а фиксируется в леджере решений как skipped.

Доставка — at-least-once: шлите смело повторно, дубликаты по event_id не сохраняются. События автоматически обновляют таймстемпы активности из раздела 6, и на них же реагирует проактивный агент (админка → «Proactive agent»; Settings → Retention bot) — поздравляет с депозитом, поддерживает после неудачного дня — в рамках жёстких ограничителей (капы, кулдауны, тихие часы, comfort-окно после проигрыша, дневной бюджет). Пока партнёрской интеграции нет, контур проверяется симулятором событий в той же админ-вкладке.

8. Тест-профиль — dev/предпрод без handshake

В dev-режиме (у продукта и деплоя не задан секрет handshake) нет хост-сайта, который подписал бы профиль, — вместо него при создании сессии подставляется тестовый профиль игрока из админки (Prompt → Prompt variables → блок Test player). Так проверяется персонализация промпта до подключения реального сайта. Как только секрет handshake задан, тест-профиль игнорируется — источником данных становится только подпись бэкенда.
