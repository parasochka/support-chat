Retention-бот «Ника» в Telegram — интеграция и API

Документация для команд разработки: как подключить второй фасад сервиса — Telegram-бот с ретеншн-персоной Никой — к сайту казино. Это тот же AI-сервис и та же мультитенантность, что у чата поддержки (обзор и архитектура — на обзорной странице; передача и синхронизация данных игрока — на странице данных игрока; авторизация и справочник /admin/* API — на странице интеграции с общей админкой); здесь описано только то, что специфично для Telegram. База URL во всех примерах — адрес этого сервиса.

1. Что это и границы

С сайта игрок уходит по диплинку в Telegram-бота, где Ника ведёт только ретеншн — живую тёплую переписку, фото и короткие видео под профиль игрока. Ядро AI — общее с чатом поддержки; отличается транспорт (Telegram webhook вместо виджета) и режим (вовлечение вместо решения вопросов).

  • Поддержку бот не обрабатывает. Любую тему про поддержку, жалобу, блок аккаунта, ввод-вывод, ответственную игру или просьбу о живом человеке Ника не решает, а роутит наружу сообщением-выбором с двумя кнопками: написать персональному менеджеру прямо в Telegram и/или перейти на сайт в чат поддержки (ссылка — per-language contact_url из Translations, а если он не задан — главная страница сайта, вычисленная из карты сайта). Показывается то, что настроено; если не настроено ничего — текстовый маршрут в поддержку.
  • Органического входа нет. Бот открывается только с сайта, всегда с данными игрока (см. диплинк). Из канала в бота попасть нельзя.
  • Всё вешается на продукт (мультитенант): у каждого казино свой бот, канал, медиатека, менеджеры, retention-KB. White-label работает из коробки, как виджет.
  • Кнопки по карте сайта. Когда Ника зовёт игрока на сайт (поиграть, пополнить, проверить баланс), под сообщением появляется одна inline-кнопка с официальной страницей из карты сайта продукта (админка → Support chat → Site map); бэкенд принимает только точный URL из этой карты. Частота периодических «напоминаний поиграть» в диалоге задаётся настройкой retention play_reminder_every_msgs (по умолчанию каждый 5-й ответ, 0 = выкл); приглашение продолжает контекст беседы и остаётся ненавязчивым.
Продукт при входящем апдейте резолвится по токену маршрутизации вебхука (аналог widget_key для виджета) — он же в пути вебхука. Профиль игрока при этом не настраивается вручную, он приходит per-player через диплинк (раздел 3).

2. Что настроить на продукте (в админке)

Всё — в админке сервиса, раздел Ретеншен (выбрав нужный продукт в шапке). Настраивается один раз на продукт; данные конкретного игрока сюда не вводятся.

ЧтоГдеСекрет?

Ранее у продукта мог быть указан только адрес Telegram-канала (или ссылка на контакт в эскалации чата поддержки). Для бота этого мало: нужно добавить именно бота — username и токен от BotFather, — и включить Retention enabled. Канал остаётся целью подписки (гейт), входом в бота он не является. Всё остальное (VIP, баланс, имя игрока) передаётся автоматически в диплинке — руками нигде не вводится.

После сохранения токена в Telegram config появляется кнопка Register / refresh webhook — она сама пропишет вебхук боту у Telegram (нужен заданный PUBLIC_BASE_URL). Проверка: https://api.telegram.org/bot<ТОКЕН>/getWebhookInfo. Бот должен быть администратором канала — иначе гейт подписки (getChatMember) не сработает.

3. Вход: диплинк и одноразовый nonce

Полный профиль игрока не влезает в start= Telegram (лимит 64 символа), поэтому обмен идёт через одноразовый nonce. Сайт (или бэкенд CMS) зовёт нашу ручку с handshake игрока и получает готовую ссылку t.me/…?start=…, которую открывает игроку.

POST/api/retention/deeplink

Тело: { widget_key?, signed_context?, escalation?, lang? }. Ответ: { nonce, deep_link }.

POST /api/retention/deeplink
{
  "widget_key": "wk_XXXXXXXXXXXX",     // тот же ключ продукта, что у виджета
  "signed_context": "<подписанный handshake игрока>",
  "escalation": false,                  // true — вход из эскалации чата поддержки
  "lang": "ru"                          // язык, на котором игрок общался (код или локаль "ru-RU")
}
→ 200 { "nonce": "r7Kx…", "deep_link": "https://t.me/nika_casino_bot?start=r7Kx…" }
  • Профиль в signed_context — тот же подписанный handshake, что и в чате поддержки (whitelisted-поля id, full_name, email, activation_status, country, balance, vip_level, registration_date). Подпись, форматы и примеры кода — на странице данных игрока. В dev (без секрета handshake) вместо подписи берётся тест-профиль из админки.
  • nonce одноразовый, живёт nonce_ttl_sec (по умолч. 120 сек) — генерируйте его в момент клика по кнопке. Повторное использование и протухший nonce отклоняются.
  • escalation различает вход: false — обычная ретеншн-кнопка (в меню только Ника); true — кнопка эскалации в чате поддержки (в меню появляется «Перейти к менеджеру»).
  • lang (опционально) — язык, на котором игрок уже общался на сайте (код ru/en/… или локаль ru-RU). Едет в nonce; при редиме бот сразу открывается на этом языке (приветствие, меню, кнопки) и разговор с Никой продолжается на нём. Без поля язык берётся из language_code клиента Telegram. Эскалация из виджета передаёт язык беседы автоматически.

При редиме на /start <nonce> бот фиксирует связку tg_user_id ↔ player_id и снимок профиля. Диплинк — единственный вход; без валидного nonce бот отвечает «открой чат через кабинет на сайте».

4. Точки входа в бота

Главный вход — кнопка эскалации в виджете поддержки, и он автоматический. Как только у продукта включён ретеншн, эскалация в чате поддержки перестаёт вести на статический contact_url и сама минтит эскалационный диплинк (escalation: true) — игрок уходит в бота, подписывается на канал и получает кнопку менеджера. На стороне сайта для этого ничего делать не нужно (виджет — основной канал); в ответе /api/chat/* у блока escalation появляется флаг retention: true, а button.url — готовый диплинк.

Дополнительно на сайте можно поставить собственные кнопки; обе зовут /api/retention/deeplink и открывают полученный deep_link (новая вкладка или tg://):

Точка входаВызовМеню в боте
Эскалация в виджете (авто, основной)минтится сервером, escalation: true, язык беседы едет сам[👤 Перейти к менеджеру] + [💬 Пообщаться с Никой]
«Написать Нике» на сайте (опц.)escalation: false[💬 Пообщаться с Никой]

// пример: клиентский обработчик кнопки «Написать Нике»
async function openNika(signedContext) {
  const r = await fetch("https://CHAT_HOST/api/retention/deeplink", {
    method: "POST", headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      widget_key: "wk_XXXXXXXXXXXX",
      signed_context: signedContext,   // подписан бэкендом CMS, TTL ≤ 120 сек
      escalation: false,
      lang: navigator.language,        // язык игрока — бот откроется на нём
    }),
  });
  const { deep_link } = await r.json();
  window.open(deep_link, "_blank");
}

5. Гейт подписки на канал

После обмена nonce, до любого меню, бот проверяет подписку на канал продукта (getChatMember) и не пускает дальше, пока игрок не подписан: показывает [📢 Открыть канал] и [✅ Я подписался] (повторная проверка). Продукт без заданного канала гейт пропускает. Бот обязан быть админом канала. Тексты гейта и кнопок редактируются в админке (Translations, ключи rtn_*) на каждый язык.

6. Профиль игрока — три уровня свежести

Набор полей — тот же whitelist, что в handshake. Схема мягко деградирует: без интеграции живёт на снапшоте, с Player API — освежается автоматически.

  1. Снапшот (готов сразу) — профиль с последнего диплинка; каждый новый диплинк его обновляет. Ноль интеграции у партнёра.
  2. Ленивый pull (когда казино реализует эндпоинт) — если на продукте задан Player API URL + ключ, сервис сам подтягивает свежий профиль перед ходом, когда снапшоту больше profile_pull_ttl_sec (по умолч. 1 ч).
  3. Push-webhook (апгрейд) — CRM казино шлёт изменения нам мгновенно (POST/partner/{product_id}/player-update, авторизация — handshake-секретом продукта, частичный апдейт).
Контракты pull (эндпоинт Player API на вашей стороне) и push (наш webhook), включая примеры запросов, — на странице данных игрока. Помимо профильных полей оба контракта принимают сигналы активности на стороне казино — опциональные ISO-8601-таймстемпы last_login_at (последний вход на сайт), last_played_at (последняя игра/ставка) и last_deposit_at (последний депозит). На них опирается резолвер состояния игрока проактивного агента (раздел 7) — статус active/at-risk/dormant, дни с последнего депозита и т.п.

7. Проактивный агент — реакции на события казино и неактивность

Единственное место, где бот пишет игроку первым, — по двум триггерам: события казино (этот раздел) и неактивность игрока (правила «молчит N дней → Ника пишет первой», лестница 7/14/30 дней, управляется в админке → Ретеншен → Proactive agent → Idle pings; те же гварды, леджеры и dry-run). Казино шлёт канонические события (POST /partner/{product_id}/event, Bearer — партнёрский секрет продукта; одно событие или батч {events:[…]} до 500, идемпотентно по event_id): депозит, крупный проигрыш, новый уровень, пройденный KYC и т.д. Фоновый воркер разбирает очередь (по умолчанию каждые 5 секунд — интервал задаётся настройкой «Worker interval» в админке), детерминированные гварды решают, можно ли вообще написать игроку, дешёвый AI-вызов решает писать / прислать фото / промолчать (молчание — полноценное решение и тоже логируется), и только затем та же AI-персона с тем же retention-KB и языком игрока пишет текст. Каждое решение попадает в аудируемый леджер (админка → «Proactive agent» → Decisions): состояние игрока, вердикт гвардов, решение, доставка, стоимость. Какие события будят агента, а какие только «кормят» состояние игрока — фиксировано встроенным набором (вкладка-редактор Triggers убрана из админки; актуальный список показывает вкладка «How it works & testing» агента). Реакция уходит не мгновенно, а со случайной задержкой 5–15 минут после события (настройки «Send delay, min/max»; мгновенная благодарность за депозит читается как слежка за транзакциями) — кнопка «Process queue now» задержку игнорирует.

Сообщение агента визуально выделено как «весточка»: над текстом идёт короткая курсивная строка-заголовок — приветствие персоны плюс человеческая фраза-повод в той же строке («✨ Привет, это Ника! Спасибо за депозит 10 USD»; ключи переводов rtn_ping_header и rtn_trig_* по событиям, редактируются по-языково в админке → Translations), а под сообщением — одна inline-кнопка по интенту («вернись поиграть» → страница игр, «пополни» → касса): модель выбирает подходящую страницу из карты сайта продукта (Support chat → Site map), бэкенд принимает только точный URL из этой карты, а прикреплённые страницы ротируются между приглашениями (казино / слоты / турниры / главная), а не повторяют одну и ту же.

Гарантии анти-спама (жёсткие, модель их не переопределяет)

  • Выключатель агента per product — настройка «Agent enabled» (Ретеншен → Настройки); dry-run по умолчанию включён: агент решает и логирует, но ничего не шлёт, пока владелец не выключит теневой режим.
  • Дневной кап на игрока (ping_daily_cap, по умолч. 3) и минимальный разрыв между любыми двумя сообщениями (ping_min_gap_hours, по умолч. 2 ч; 0 = выключен) — сколько бы событий ни пришло.
  • Кулдаун по типу события (v2_same_event_cooldown_hours, по умолч. 5 ч) — пять депозитов за вечер получают одну тёплую весточку, не пять. 0 = выключен (режим тестов).
  • Тихие часы по местному времени аудитории (quiet_hours_start=22, quiet_hours_end=9, quiet_hours_utc_offset=0) — ночью сообщения не уходят.
  • Дневной AI-бюджет (v2_daily_budget_usd) — достигнут ⇒ агент молчит до завтра.
  • Окно утешения после проигрыша (v2_loss_comfort_hours / v2_loss_high_usd) — после крупного проигрыша: только эмпатия, без фото, без ссылок, без приглашений играть.
  • Опт-аут игрока: команда /stop в боте отключает проактив (вернуть — /resume); реактивные ответы Ники при этом продолжают работать.
  • Заблокировавших бота не дёргаем: игрок, заблокировавший бота, помечается недоступным и пропускается, пока сам не напишет снова.
  • Ограниченный батч за проход (ping_batch_size, по умолч. 30) — защита от всплеска расходов.

Все пороги — в hot-reload-группе настроек retention (пер-продукт; Settings → Retention bot → «Proactive agent» и «Send-frequency guards»), деплой-выключатель воркера — env (RETENTION_SCHEDULER_ENABLED, см. README). Разбор очереди атомарный (claim через FOR UPDATE SKIP LOCKED): воркер, кнопка «Process queue now» в админке и второй инстанс сервиса могут работать одновременно — одно событие никогда не даст двух сообщений. В админке есть симулятор событий и кнопка ручного прогона для тестов до подключения казино.

9. Переменные окружения (деплой)

ПеременнаяНазначение
