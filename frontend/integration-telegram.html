Retention-бот «Ника» в Telegram — интеграция и API

Документация для команд разработки: как подключить второй фасад сервиса — Telegram-бот с ретеншн-персоной Никой — к сайту казино. Это тот же AI-сервис и та же мультитенантность, что у чата поддержки (см. страницу интеграции чата поддержки); здесь описано только то, что специфично для Telegram. База URL во всех примерах — адрес этого сервиса.

1. Что это и границы

С сайта игрок уходит по диплинку в Telegram-бота, где Ника ведёт только ретеншн — живую тёплую переписку и фото под профиль игрока. Ядро AI — общее с чатом поддержки; отличается транспорт (Telegram webhook вместо виджета) и режим (вовлечение вместо решения вопросов).

  • Поддержку бот не обрабатывает. Любую тему про поддержку, жалобу, блок аккаунта, ввод-вывод, ответственную игру или просьбу о живом человеке Ника не решает, а роутит наружу: эскалированный вход → к менеджеру, ретеншн-вход → назад в поддержку на сайте.
  • Органического входа нет. Бот открывается только с сайта, всегда с данными игрока (см. диплинк). Из канала в бота попасть нельзя.
  • Всё вешается на продукт (мультитенант): у каждого казино свой бот, канал, медиатека, менеджеры, retention-KB. White-label работает из коробки, как виджет.
Продукт при входящем апдейте резолвится по токену маршрутизации вебхука (аналог widget_key для виджета) — он же в пути вебхука. Профиль игрока при этом не настраивается вручную, он приходит per-player через диплинк (раздел 3).

2. Что настроить на продукте (в админке)

Всё — в админке сервиса, раздел Retention · Telegram (выбрав нужный продукт в шапке). Настраивается один раз на продукт; данные конкретного игрока сюда не вводятся.

ЧтоГдеСекрет?
Bot username (например nika_casino_bot) + Bot token от BotFather Telegram config → поле + Secretsтокен — да (шифруется, наружу только флаг)
Channel id (@channel или -100…) + Channel URL Telegram configнет

Ранее у продукта мог быть указан только адрес Telegram-канала (или ссылка на контакт в эскалации чата поддержки). Для бота этого мало: нужно добавить именно бота — username и токен от BotFather, — и включить Retention enabled. Канал остаётся целью подписки (гейт), входом в бота он не является. Всё остальное (VIP, баланс, имя игрока) передаётся автоматически в диплинке — руками нигде не вводится.

После сохранения токена в Telegram config появляется кнопка Register / refresh webhook — она сама пропишет вебхук боту у Telegram (нужен заданный PUBLIC_BASE_URL). Проверка: https://api.telegram.org/bot<ТОКЕН>/getWebhookInfo. Бот должен быть администратором канала — иначе гейт подписки (getChatMember) не сработает.

3. Вход: диплинк и одноразовый nonce

Полный профиль игрока не влезает в start= Telegram (лимит 64 символа), поэтому обмен идёт через одноразовый nonce. Сайт (или бэкенд CMS) зовёт нашу ручку с handshake игрока и получает готовую ссылку t.me/…?start=…, которую открывает игроку.

POST/api/retention/deeplink

Тело: { widget_key?, signed_context?, escalation? }. Ответ: { nonce, deep_link }.

POST /api/retention/deeplink
{
  "widget_key": "wk_XXXXXXXXXXXX",     // тот же ключ продукта, что у виджета
  "signed_context": "<подписанный handshake игрока>",
  "escalation": false                   // true — вход из эскалации чата поддержки
}
→ 200 { "nonce": "r7Kx…", "deep_link": "https://t.me/nika_casino_bot?start=r7Kx…" }
  • Профиль в signed_context — тот же подписанный handshake, что и в чате поддержки (whitelisted-поля id, full_name, email, activation_status, country, balance, vip_level, registration_date). Подпись, форматы и примеры кода — на странице интеграции чата поддержки, раздел 3. В dev (без секрета handshake) вместо подписи берётся тест-профиль из админки.
  • nonce одноразовый, живёт nonce_ttl_sec (по умолч. 120 сек) — генерируйте его в момент клика по кнопке. Повторное использование и протухший nonce отклоняются.
  • escalation различает вход: false — обычная ретеншн-кнопка (в меню только Ника); true — кнопка эскалации в чате поддержки (в меню появляется «Перейти к менеджеру»).

При редиме на /start <nonce> бот фиксирует связку tg_user_id ↔ player_id и снимок профиля. Диплинк — единственный вход; без валидного nonce бот отвечает «открой чат через кабинет на сайте».

4. Две кнопки на сайте

На сайте казино ставятся две точки входа; обе зовут /api/retention/deeplink и открывают полученный deep_link (новая вкладка или tg://):

КнопкаВызовМеню в боте
«Написать Нике» (ретеншн)escalation: false[💬 Пообщаться с Никой]
Эскалация в чате поддержкиescalation: true[👩 К менеджеру] + [💬 Ника]
// пример: клиентский обработчик кнопки «Написать Нике»
async function openNika(signedContext) {
  const r = await fetch("https://CHAT_HOST/api/retention/deeplink", {
    method: "POST", headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      widget_key: "wk_XXXXXXXXXXXX",
      signed_context: signedContext,   // подписан бэкендом CMS, TTL ≤ 120 сек
      escalation: false,
    }),
  });
  const { deep_link } = await r.json();
  window.open(deep_link, "_blank");
}

5. Гейт подписки на канал

После обмена nonce, до любого меню, бот проверяет подписку на канал продукта (getChatMember) и не пускает дальше, пока игрок не подписан: показывает [📢 Открыть канал] и [✅ Я подписался] (повторная проверка). Продукт без заданного канала гейт пропускает. Бот обязан быть админом канала. Тексты гейта и кнопок редактируются в админке (Translations, ключи rtn_*) на каждый язык.

6. Профиль игрока — три уровня свежести

Набор полей — тот же whitelist, что в handshake. Схема мягко деградирует: без интеграции живёт на снапшоте, с Player API — освежается автоматически.

  1. Снапшот (готов сразу) — профиль с последнего диплинка; каждый новый диплинк его обновляет. Ноль интеграции у партнёра.
  2. Ленивый pull (когда казино реализует эндпоинт) — если на продукте задан Player API URL + ключ, сервис сам подтягивает свежий профиль перед ходом, когда снапшоту больше profile_pull_ttl_sec (по умолч. 1 ч).
  3. Push-webhook (апгрейд) — CRM казино шлёт изменения нам мгновенно.

Pull — контракт на стороне казино (реализуете вы у себя)

GET  {player_api_url}?player_id=<id>
Authorization: Bearer <player_api_key>
→ 200 {
  "id": "player-123", "full_name": "Иван Петров", "email": "ivan@example.com",
  "activation_status": "active", "country": "Portugal", "balance": "1500.00 EUR",
  "vip_level": "Silver", "registration_date": "2024-01-15"
}

Push — контракт на нашей стороне (шлёте вы при изменении профиля)

POST/partner/{product_id}/player-update
POST /partner/42/player-update
Authorization: Bearer <handshake-секрет продукта>
{ "player_id": "player-123", "balance": "2000.00 EUR", "vip_level": "Gold" }
→ 200 { "ok": true, "updated": 1 }

Авторизация push — секретом handshake продукта (тот же общий секрет, что подписывает handshake). Отсутствующие поля не трогаются (частичный апдейт).
